# i18n Design — Russian & Spanish Support

**Date:** 2026-04-21
**Scope:** Full app (all pages and components)
**Languages:** English (en), Russian (ru), Spanish (es)

---

## Overview

Add multilingual support to Wondish using `next-intl`. Language is stored in a `NEXT_LOCALE` cookie (1-year expiry) so the user's choice persists across sessions. URLs do not change — locale is resolved purely via middleware and cookie. Default locale is English.

---

## Architecture

### Dependencies
- `next-intl` — the only new dependency

### Middleware (`middleware.ts`)
- Runs on every request
- Reads `NEXT_LOCALE` cookie, defaults to `en` if absent
- Sets locale context for the request via `next-intl`'s `createMiddleware` (or manual approach with `unstable_setRequestLocale`)

### Message files
```
messages/
  en.json
  ru.json
  es.json
```
Organized by namespace. All three files share identical key structure.

### Provider setup
`app/layout.tsx` wraps children in `NextIntlClientProvider` with the messages for the active locale, resolved server-side via `getLocale()`.

### Usage in components
- Server components: `const t = await getTranslations('namespace')`
- Client components: `const t = useTranslations('namespace')`

---

## Language Switcher

Location: Navbar (desktop top-right, mobile menu)

UI: A small dropdown showing current locale as flag + code:
- 🇬🇧 EN
- 🇷🇺 RU
- 🇪🇸 ES

Behavior: Selecting a language calls `POST /api/set-locale` with `{ locale: 'ru' }`, which sets the `NEXT_LOCALE` cookie (max-age 1 year) and returns success. The client then does `router.refresh()` to re-render with the new locale.

## Constraints & Notes

- No URL changes — locale is cookie-only, not path-based
- `next-intl` requires `setRequestLocale(locale)` in every server page/layout that uses translations (Next.js 14 App Router requirement for static rendering)
- Admin pages are translated (full app scope)
- Chinese is out of scope for this implementation
- Translation strings for RU and ES are written directly into the message files (no external translation service)
- Dynamic data (recipe names, ingredient names from DB) is out of scope — only UI strings are translated
